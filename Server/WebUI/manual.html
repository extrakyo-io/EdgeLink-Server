01產品概述

EdgeLink Server 是一個輕量化的 IoT 協定橋接器，執行於 Windows 上，無需安裝 .NET Runtime。核心功能是讓 TCP Server（設備端）收到的訊息，透過 Mask（資料轉換規則）處理後，自動路由到 TCP Client（系統端），回傳方向亦同。所有設定都可在瀏覽器介面即時操作，無需重啟服務。

02安裝與啟動

  1. 從 Releases 下載 EdgeLink-Server-win-x64.zip 並解壓縮到任意目錄
  2. 執行 EdgeLinkServer.exe（雙擊或在命令提示字元中執行）
  3. 開啟瀏覽器前往 https://localhost:8443
  4. 輸入預設密碼 admin 登入，登入後立即修改密碼
憑證警告 首次使用 HTTPS 時瀏覽器會顯示憑證警告（自簽憑證）。點「進階 → 繼續前往」即可正常使用。憑證已包含本機所有 IP（SAN），區域網路內的其他電腦同樣只需接受一次警告。

常用啟動參數

CMD 
# 預設（HTTPS，port 8443）
EdgeLinkServer.exe

# 僅 HTTP
EdgeLinkServer.exe --no-https

# 自訂 port

06PING/PONG 心跳協定

設備端必須實作 EdgeLink 會對每個連入的設備定期發送心跳。設備端程式必須回應 PONG，否則約 15～20 秒後被強制斷線。
例外：二進位埠不走心跳 若該埠套用的是 Binary Mask，EdgeLink 不會發送 EDGELINK_PING（避免文字混進二進位串流），來源端也不需要回 PONG。斷線改由 TCP keep-alive（閒置 10 秒、每秒探測）偵測。使用官方 SDK 則兩種情況都已自動處理。
首次心跳
3s
連線後發送
心跳間隔
5s
持續發送
斷線閾值
3次
連續未回應即斷線

訊息格式

方向格式範例
EdgeLink → 設備EDGELINK_PING:<16位元16進位>\nEDGELINK_PING:a3f7c291e05d8b4a\n
設備 → EdgeLinkEDGELINK_PONG:<相同16進位>\nEDGELINK_PONG:a3f7c291e05d8b4a\n

Python 實作範例

Python
import socket, threading

def receive_loop(sock):
    buf = b""
    while True:
        data = sock.recv(4096)
        if not data:
            break
        buf += data
        while b"\n" in buf:
            line, buf = buf.split(b"\n", 1)
            msg = line.decode().strip()
            if msg.startswith("EDGELINK_PING:"):
                hex_val = msg.split(":")[1]
                sock.sendall(f"EDGELINK_PONG:{hex_val}\n".encode())
            elif not msg.startswith("EDGELINK_"):
                handle_message(msg)  # 處理業務邏輯

sock = socket.socket()
sock.connect(("127.0.0.1", 9001))
threading.Thread(target=receive_loop, args=(sock,), daemon=True).start()
sock.sendall(b"id:DEV01;value:36.5\n")
常見陷阱：socket 逾時會造成「假斷線」 上例刻意用 socket.socket() 讓 recv 保持阻塞。若改用 socket.create_connection(addr, timeout=5)，那個逾時連線後仍會留在 socket 上；由於心跳每 5 秒才來一次，recv 會在兩次 PING 之間丟出 socket.timeout，若被誤當成斷線就會每 5～15 秒重連一次。連線後請補上 sock.settimeout(None)。

07Mask 系統

Mask 定義如何將設備的原始訊息解析並轉換成新格式，再轉發給系統端（或反向）。

設定位置 Request Mask 和 Response Mask 都設定在 TCP Client Port 上，不是 TCP Server Port。
設備送出 "id:DEV01;temp:36.5"
Request Mask ← TCP Client 的 Mask Type
轉換後
系統端收到 {"deviceId":"DEV01","temperature":36.5}
系統端回覆 "status:ok;code:200"
Response Mask ← TCP Client 的 Response Mask Type
轉換後
設備收到 "OK:ok"

Mask 定義欄位

欄位說明預設值
Mask ID唯一識別名稱—
Field Delimiter欄位之間的分隔符;
KV Separator鍵與值之間的分隔符:
Output Template輸出模板，以 {欄位名} 作為佔位符—
Sample Data範例輸入，用於即時預覽—
Route Modebroadcast / response（控制回應發送對象）broadcast
Correlation ID FieldConcurrent 模式：回應中帶有 correlationId 的欄位名—

欄位解析範例

輸入：id:DEV01;temp:36.5;unit:C（分隔符 ;、KV 分隔 :）

欄位名解析值
idDEV01
temp36.5
unitC

模板：{"id":"{id}","temp":{temp},"unit":"{unit}"}
輸出：{"id":"DEV01","temp":36.5,"unit":"C"}

模板規則

  • 佔位符格式：{欄位名}
  • 任何一個佔位符對應的欄位在訊息中不存在 → 整筆訊息丟棄，不轉發
  • 模板為空或 {raw} → 原樣轉發，不做任何轉換
  • 同一欄位可在模板中出現多次

Route Mode（回應路由模式）

模式行為適用場景
broadcast（預設）所有已連線的設備都收到回應廣播狀態更新、感測器推播
response只有發出此次請求的設備收到回應Request-Response 模式、設備獨立查詢

08Binary Mask（二進位解碼）

一般 Mask 處理的是文字（key:value）。若設備送的是原始二進位協定，可改用 Binary Mask：EdgeLink 直接依位元組版面解碼，轉成 KV 文字再轉發 —— 下游（Unity／儀表板）收到的是純文字，完全看不到二進位。

解碼發生在「來源埠」 把 Binary Mask 設在接收資料的那個埠（TCP Server 或 UDP）。轉發／輸出埠請設 OriginalData，原樣把已解碼的 KV 送出去，不要再套一次。

BinarySpec 結構

欄位說明預設值
byteOrder多位元組欄位的位元組序：little / biglittle
syncTCP 專用：封包起始的 magic（hex，如 4f4b = OK），用來對齊分包與雜訊後重新同步。UDP 免填（空）
discriminator讀哪個位址的值來挑 variant，例如訊息型別在 offset 3：{ "offset": 3, "type": "u8" }—
variants各種封包型別的版面定義（見下）—

variant 欄位

欄位說明
matchdiscriminator 的值等於此數才套用這個 variant
isDefaulttrue = 不比對、永遠符合（沒有 discriminator 時用）
length此型別封包的固定長度；長度不符即丟棄。TCP 分包也靠它決定要收多少位元組
template輸出的 KV 模板，用 {欄位名} 佔位；缺任一欄位則整筆丟棄
fields欄位清單（見下）

field 型別

type說明
u8 u16 u32 u64無號整數（1／2／4／8 bytes）
i8 i16 i32 i64有號整數
f32 f64IEEE 754 浮點
bit取 offset 該位元組的第 bit 位（0 = LSB），輸出 0／1
bitrange從第 bit 位取 count 個位元，組成一個整數
const不讀封包，直接輸出 value 字串（常用來塞裝置 id）

數值型另可加 scale / add（輸出 = 原值 × scale + add）與 format（如 0.###）。

UDP 與 TCP 的差別

傳輸分包方式心跳
UDP每個 datagram 就是一個完整封包，不需要 sync無（以逾時判斷 stale）
TCP串流沒有封包邊界 → 必須設 sync，EdgeLink 依「magic 對齊 → discriminator 查長度 → 湊滿才解碼」切包，殘缺會等下一段、雜訊會自動重新同步不送 PING/PONG，改用 TCP keep-alive
走 TCP 一定要設 sync 沒有 magic 就無法從雜訊或半截封包中重新對齊，一旦錯位就會持續解碼失敗。UDP 則不受影響。
二進位埠不需要回心跳 為了避免文字 PING 混進二進位串流，EdgeLink 不會對二進位埠送 EDGELINK_PING，來源端自然也不必回 PONG。斷線偵測改由 TCP keep-alive（閒置 10 秒、每秒探測）負責。

設定範例

同一個埠可依 msgType（offset 3）分派到不同版面：

JSON 
{
  "byteOrder": "little",
  "sync": "4f4b",
  "discriminator": { "offset": 3, "type": "u8" },
  "variants": [
    {
      "match": 1,
      "length": 37,
      "template": "id:{id};seq:{seq};x:{x};fault:{fault}",
      "fields": [
        { "name": "id",    "offset": 0,  "type": "const", "value": "dev1" },
        { "name": "seq",   "offset": 5,  "type": "u32" },
        { "name": "x",     "offset": 19, "type": "f32", "format": "0.###" },
        { "name": "fault", "offset": 27, "type": "bitrange", "bit": 0, "count": 2 }
      ]
    }
  ]
}

解碼後轉發出去的就是：id:dev1;seq:1234;x:0.512;fault:0

可在 Mask 分頁用二進位版面編輯器建立（附 hex 解碼即時預覽），或用 設定匯入整包帶進來。

09Modbus TCP Master

EdgeLink 內建 Modbus TCP master，可定時輪詢 Modbus slave（ICP DAS ET-7x00、一般 PLC、模擬器等），把讀到的暫存器組成 KV 訊息送進路由管線。下游消費端不會知道資料來源是 Modbus 還是一般 TCP。

欄位說明範例
Slave IPModbus slave 位址192.168.1.10
Modbus Port通訊埠502
Slave ID從站編號（Unit ID）1
Polling Interval輪詢間隔（毫秒）100（10 Hz）
Device ID會變成輸出訊息裡的 id: 欄位rig1
Registers暫存器對應表（JSON 陣列，見下）—

Function Code

FCModbus 操作典型用途
1Read Coils輸出位元（繼電器、LED 狀態）
2Read Discrete Inputs開關、按鈕、編碼器位元
3Read Holding Registers一般 16-bit 可讀寫暫存器

資料型別

dataType長度說明
bit1 bit單一 Coil／DI（quantity 必須為 1）
bitsN bits把 quantity 個位元打包成一個無號整數（LSB first）
uint16 / int162 bytes單一暫存器
uint32 / int324 bytes兩個暫存器，big-endian

暫存器對應範例

JSON 
[
  { "name": "yaw",   "functionCode": 2, "startAddress": 0, "quantity": 8, "dataType": "bits" },
  { "name": "pitch", "functionCode": 2, "startAddress": 8, "quantity": 8, "dataType": "bits" },
  { "name": "joyx",  "functionCode": 4, "startAddress": 0, "quantity": 1, "dataType": "uint16", "scale": 0.001 },
  { "name": "joyy",  "functionCode": 4, "startAddress": 1, "quantity": 1, "dataType": "uint16", "scale": 0.001 }

每個輪詢週期會產生一筆合成訊息：id:rig1;yaw:128;pitch:64;joyx:2.5;joyy:1.802

輪詢頻率要配合模組掃描率 設得比模組實際掃描率快並不會拿到更新的值，只會增加負載。類比輸入模組（單 ADC 輪掃）常見上限只有數 Hz，數位輸入則快得多。

11請求模式

Request Mode 設定在 TCP Client Port 上，控制如何處理同時進來的多筆請求。

S
Serial
序列模式

FIFO 佇列，一次處理一筆，等到收到回應（或 5 秒 timeout）後才處理下一筆。不遺漏任何訊息。

預設模式 不遺漏訊息
P
Polling
輪詢模式

只保留最新一筆資料，中間的請求會被丟棄。適合持續傳送感測數值、只需最新值的場景。

會丟棄舊訊息
C
Concurrent
並發模式

多個請求同時進行，EdgeLink 自動注入 _corrId，依 ID 將回應路由回正確的設備。

需設定 correlationId

Concurrent 模式設定步驟

  1. 建立 Request Mask，模板中包含 {_corrId}（EdgeLink 自動注入）：
    Template
    {data}:{_corrId}
  2. 建立 Response Mask，設定 Route Mode: response，Correlation ID Field 填入回應中帶有 correlationId 的欄位名（例如 _corrId）。
  3. TCP Client Port 設定：Request Mode: concurrent，Mask Type 和 Response Mask Type 分別選剛才建立的兩個 Mask。
  4. 遠端系統回應時必須帶回相同的 correlationId 欄位。

Concurrent 模式流程圖

設備 A 送出 → _corrId=AAAA → 遠端系統
設備 B 送出 → _corrId=BBBB → 遠端系統
設備 C 送出 → _corrId=CCCC → 遠端系統
同時進行 → 各自回應
遠端回應 → _corrId=BBBB → 路由回設備 B
遠端回應 → _corrId=AAAA → 路由回設備 A
遠端回應 → _corrId=CCCC → 路由回設備 C

設備連線 / 斷線偵測

EdgeLinkManager 提供兩種互補的斷線偵測機制，可同時使用：

事件觸發來源識別單位適用情境
OnDeviceStatus(bool, string) EdgeLink Server 偵測到 TCP 連線開/關 IP 位址 快速感知連線層斷線（含拔電約 15 秒）
OnDeviceTimeout(string) Unity 本地計時，超過 Device Timeout 秒無訊息 設備 ID 欄位 多台設備共用一條連線，精確追蹤個別設備
OnDeviceReconnected(string) 超時後再次收到該設備訊息 設備 ID 欄位 設備重新上線通知
C# — 斷線偵測範例
void Start()
{
    edgeLink = GetComponent<EdgeLinkManager>();

    // TCP 連線層事件（由 EdgeLink Server 推送）
    edgeLink.OnDeviceStatus += (connected, endpoint) =>
        Debug.Log(connected ? $"設備上線: {endpoint}" : $"設備斷線: {endpoint}");

    // 業務層事件（Unity 本地計時，依 Device Id Key 欄位追蹤）
    edgeLink.OnDeviceTimeout     += id => Debug.LogWarning($"{id} 離線（超時）");
    edgeLink.OnDeviceReconnected += id => Debug.Log($"{id} 重新上線");
}
拔電斷線情境 設備突然斷電（無 FIN 封包）時，EdgeLink 透過 PING/PONG 機制偵測：連續 3 次 PING 無回應（約 15 秒）後，Server 強制關閉連線並推送 OnDeviceStatus(false, …)。之後若超過 Device Timeout 秒仍無訊息，OnDeviceTimeout 也會觸發。兩個事件一起使用可獲得最完整的斷線資訊。

TCP 模式拓樸（推薦）

IoT 設備（TCP 連入 EdgeLink）
TCP Server Port 9001
EdgeLink Server（TCP Client 主動連 Unity）
TCP Client → Unity Host:Port
Unity 應用程式（EdgeLinkManager Protocol = TCP）

EdgeLink Server 端需設定一個 TCP Client Port，Target IP 填 Unity 機器 IP，Remote Port 對應 TCP Port 欄位值，Source Protocol 指向接收設備的 TCP Server Port。

TCP 模式（ESP32 / ESP8266）

設備主動連線到 EdgeLink Server 的 TCP Server Port，EdgeLink 自動管理 PING/PONG。

Arduino C++ — tcp_sensor.ino
#include <WiFi.h>
#include <EdgeLink.h>

WiFiClient  wifiClient;
EdgeLinkTCP edgelink(wifiClient);

void onMessage(const String& msg) {
    Serial.println(msg);  // 後端系統的回應
}

void setup() {
    Serial.begin(115200);
    WiFi.begin("your-ssid", "your-password");
    while (WiFi.status() != WL_CONNECTED) delay(500);

    edgelink.onMessage(onMessage);
    edgelink.setAutoReconnect(true, 5000);   // 斷線後 5 秒重連
    edgelink.begin("192.168.1.100", 9001);  // EdgeLink TCP Server Port
}

void loop() {
    edgelink.loop();  // 必須呼叫，處理 PING/PONG 與接收

    static uint32_t t = 0;
    if (millis() - t >= 3000 && edgelink.isConnected()) {
        edgelink.send("id:ESP32_01;temp:25.3;humidity:60.0");
        t = millis();
    }
}

UDP 模式（ESP32 / ESP8266）

無連線狀態，每次直接傳送 UDP 封包到 EdgeLink Server 的 UDP Port。EdgeLink 不對 UDP 設備發送 PING/PONG。

Arduino C++ — udp_sensor.ino
#include <WiFi.h>
#include <WiFiUdp.h>
#include <EdgeLink.h>

WiFiUDP     wifiUdp;
EdgeLinkUDP edgelink(wifiUdp);

    Serial.println(msg);
}

void setup() {
    Serial.begin(115200);
    WiFi.begin("your-ssid", "your-password");
    while (WiFi.status() != WL_CONNECTED) delay(500);

    edgelink.begin(4210);          // 本機接收 Port
    edgelink.onMessage(onMessage);
}

void loop() {
    edgelink.loop();  // 接收 UDP 封包

    static uint32_t t = 0;
    if (millis() - t >= 3000) {
        edgelink.send("192.168.1.100", 9002, "id:ESP32_01;temp:25.3;humidity:60.0");
        t = millis();
    }
}

21常見情境範例

情境 A：感測器 → 後端 JSON API（單向轉發）

設備傳送： id:SENSOR01;temp:25.3;humidity:60
目標： 後端收到 {"device":"SENSOR01","temp":25.3,"humidity":60}

  1. 建立 Mask SensorJSON：
    Mask 設定
    Field Delimiter : ;
KV Separator    : :
Output Template : {"device":"{id}","temp":{temp},"humidity":{humidity}}
  2. 新增 TCP Server Port：Protocol Name: SensorServer，Local Port: 9001
  3. 新增 TCP Client Port：Protocol Name: BackendClient，Target IP: 192.168.1.100，Remote Port: 8080，Mask Type: SensorJSON，Source Protocol: SensorServer

情境 B：多台設備同時查詢各自收到回應（Concurrent）

場景： 多台設備同時送出請求，每台只收到自己的回應。

  1. 建立 Request Mask，模板包含 {_corrId}：{barcode}:{_corrId}
  2. 建立 Response Mask，Route Mode: response，Correlation ID Field: _corrId
  3. TCP Client Port 設定 Request Mode: concurrent
  4. 後端系統解析 correlationId，回應時帶回同一個值

情境 C：設備查詢並等待個別回應（Serial Request-Response）

  1. TCP Client Port：Request Mode: serial
  2. Response Mask：Route Mode: response
  3. 後端在 5 秒內於同一 TCP 連線上回覆，EdgeLink 自動路由回對應的設備

情境 D：同一組設備廣播到多個系統（一對多）

  1. 建立一個 TCP Server Port（設備連入）
  2. 為每個系統建立一個 TCP Client Port，Source Protocol 全部指向同一個 TCP Server
  3. 每個 TCP Client 可設定不同的 Mask 規則（不同系統不同格式）

22疑難排解

設備連線後約 15 秒斷線

原因：未實作 PING/PONG EdgeLink 連續 3 次收不到 PONG 後強制斷線。請在設備程式中加入接收 EDGELINK_PING: 並回覆 EDGELINK_PONG: 的邏輯（詳見第 6 節）。

瀏覽器無法連線

情況解法
剛啟動，程式還在初始化等待 3～5 秒後重新整理
Port 已被其他程式佔用用 --port 換一個 port
防火牆阻擋允許 EdgeLinkServer.exe 或對應 port 通過防火牆
LAN 存取被拒以系統管理員身分執行

訊息沒有被路由到後端

  • TCP Client Port 的 Source Protocol 是否正確指向 TCP Server
  • TCP Client Port 是否顯示已連線
  • Mask 設定：訊息是否包含模板中所有的佔位符欄位（缺一個就整筆丟棄）
  • 在 Monitor 頁面即時查看訊息是否有進來

後端回應沒有送回設備

  • broadcast 模式：確認設備是否真的有連線
  • response 模式（Serial）：5 秒內沒收到回應會 timeout
  • response 模式（Concurrent）：後端是否有帶回正確的 correlationId
  • Response Mask 的欄位是否都能被正確解析

TCP Client 一直顯示未連線

  • 確認遠端系統的 IP 和 port 是否正確
  • 確認遠端系統是否正在監聽
  • EdgeLink 會自動重連：首次等待 1 秒，之後指數退避最長 30 秒
